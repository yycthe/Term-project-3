Topic 
Our group genuinely likes the NBA—we talk about matchups, momentum, and “who has the edge” all the time—so we wanted to turn that fan-level intuition into something testable, auditable, and useful. The project goal is to predict NBA game outcomes before tip-off while producing a calibrated win probability, not just a win/lose label.[5] That probability output matters because it supports real decisions: comparing matchups, tracking confidence, and doing honest post-game reviews of where we were wrong. The task is challenging because sports prediction is noisy and non-stationary, and it’s easy to “accidentally cheat” through data leakage (e.g., using information that is only known after the game).[2] For that reason, the project is designed as an end-to-end, leakage-aware forecasting system rather than a one-off classifier.[2][3][4] Beyond a fixed pipeline, we built a closed-loop, agentic workflow that repeatedly observes results, adapts its search strategy based on evidence, and finalizes deployable artifacts: it automatically engineers pre-game features (rolling stats, Elo, matchup differences), searches model/feature configurations with Optuna, reflects to narrow the search space for the next phase, and deploys results in a UI so we can use it the way we actually watch and analyze games.[1][6][7] 

AI Technology Stack 
We use an end-to-end AI stack across data, modeling, optimization, and deployment to make the system practical for real NBA fans and real forecasting constraints. On the data side, we rely on Pandas/Numpy for ingestion and transformation and nba_api/requests to refresh real NBA game logs, with explicit leakage detection to remove post-game information that does not belong in a pre-tip prediction pipeline.[2] On the modeling side, we use scikit-learn for strong baselines and classical models (Logistic Regression, Random Forest) and XGBoost for gradient-boosted trees that handle non-linear interactions common in matchup data. Because our output is a win probability, we include probability calibration (e.g., isotonic calibration for tree-based models) to make predicted probabilities more trustworthy.[5] We evaluate models with probability-centered metrics such as LogLoss and Brier score, while also tracking AUC and Accuracy for discrimination and interpretability.[4] For automated improvement, we integrate Optuna (TPE sampler) as the agent’s search engine, where each trial samples a full decision package (window_years, elo_k, model family, feature strategy, top_k, feature-type switches, and model hyperparameters).[6] The agent also maintains cross-run memory (agent_memory.json) and produces auditable policies (outputs/policy.json) to narrow search ranges over time, and it can trigger diagnostic actions (e.g., ablations) and write reports (outputs/diagnostics.md) when performance plateaus.[6] Finally, we deploy the pipeline through a Streamlit UI that loads serialized artifacts (joblib) and supports single-model and ensemble predictions, turning our work into a usable application rather than a disconnected notebook.[7] 

AI Model Selection 
Our model choices match the nature of the problem: predicting pre-game outcomes from engineered, tabular features is a mixed-signal classification task where both linear baselines and tree-based methods are appropriate. We include Logistic Regression as a stable, interpretable baseline that helps validate whether the feature pipeline is capturing meaningful basketball signals and provides a reliable reference point. We then use Random Forest and XGBoost, which are strong for tabular sports data because they capture non-linear effects and interactions—such as Elo difference interacting with recent form, rest days, and team efficiency—without requiring heavy feature scaling.[3] Crucially, we treat this as a probability prediction problem, so we apply probability calibration when needed and prioritize probability-quality metrics (LogLoss/Brier) in model comparison, not just classification accuracy.[4][5] Models are compared under a consistent, season-aware evaluation setup, and when diverse top performers are complementary, we blend them into a simple ensemble to reduce variance and improve the stability of win probabilities—especially important in a high-noise domain like sports.[4] 

AI Model Training / Validation 
Our training and validation design follows a realistic forecasting scenario and directly addresses the most common failure modes in sports prediction, while also implementing an agentic closed-loop rather than a single-pass training script.[1] Instead of random splitting, we use a season-aware time split: the model trains on historical seasons and is evaluated on the most recent season, so the test set truly represents “future” games the model has not seen.[4] Feature engineering is explicitly leakage-safe: rolling team statistics are computed chronologically and shifted (e.g., shift(1)) so each game’s features only use information available before that game, and a leakage detector removes post-game box-score columns (points, rebounds, shooting percentages, etc.).[2][3] Within this pipeline, the agent operates as OBSERVE → PLAN → EXPLORE → REFLECT/ADAPT → EXPLOIT → DIAGNOSE → FINALIZE.[1] In OBSERVE, it refreshes data when possible, loads the dataset, detects key columns (date/teams/target), and performs leakage checks.[2] In PLAN, it optimizes an explicit objective—minimize LogLoss under an AUC floor constraint—while reading cross-run memory to warm-start and guide the next run.[1] In EXPLORE, Optuna runs Phase 1 with broader ranges to learn what works; then in REFLECT/ADAPT, the agent analyzes top trials and writes an auditable policy (outputs/policy.json) that narrows the search space (e.g., window_years, elo_k, feature-type patterns).[6] In EXPLOIT, Phase 2 runs in the narrowed space to refine performance, optionally enqueuing the current best configuration to guide the sampler early.[6] If results are stuck above target, DIAGNOSE triggers actions beyond tuning, such as ablation studies, and produces a diagnostic report (outputs/diagnostics.md).[6] In FINALIZE, the system saves the best model, preprocessors, calibration artifacts, ensemble members, and run reports so the Streamlit UI can load them and the full workflow can be audited later.[1][7] To ensure probability reliability, tree-based models are calibrated using isotonic calibration, and calibration curves are generated to verify whether predicted probabilities match observed win rates.[4][5] 

AI Model Performance Criteria 
We use different criteria to evaluate different model families under the same season-based test setup, with an emphasis on probability reliability because that is what our group actually wants to use before tip-off.[4] LogLoss is the primary metric because it directly rewards well-calibrated probabilities and strongly penalizes overconfident mistakes.[4] We also report Brier score as an additional probability-accuracy measure, along with AUC for discrimination and Accuracy as an intuitive reference at the 0.5 threshold.[4] To avoid “gaming” LogLoss by producing weak rankings, we enforce an AUC floor constraint during tuning: trials below the minimum AUC threshold are penalized and effectively rejected.[1] This creates a balanced selection rule—choose the model (or ensemble) that produces the best probability quality (low LogLoss/Brier) while maintaining acceptable discrimination (AUC)—so the final win probabilities are both accurate and useful for real matchup analysis.[4]

Recommendations  
First, we should treat the model as a probability tool, not a “sure winner” picker. In real use, we can define a simple decision rule: only take action when the predicted win probability is meaningfully high (for example, Win% ≥ 60%), and label anything around 50–55% as a “coin flip.” This directly fits the business goal—making decisions under uncertainty—while avoiding over-trusting tiny edges that are basically noise. Second, we should add a calibration KPI so we can measure whether “Win% matches reality,” not just eyeball calibration curves. Keeping calibration plots is good, but we should also report a numeric metric such as ECE or a simple binned accuracy table, because the business value depends on statements like “70% should win about 70% of the time.” Third, we should make evaluation more robust by adding multi-season backtesting with rolling season splits. That would let us claim the system generalizes over time, instead of only performing on one latest-season test, which increases trust if we actually deploy it. Fourth, we should capture real-world pre-game factors that box-score-style features often miss, such as back-to-back games, 3-in-4 nights, travel distance/time zones, and (if available) injury/availability signals; these are exactly the kinds of signals that explain many high-confidence misses. Fifth, we should improve fairness and stability in model selection by ensuring each model family receives comparable tuning effort—for example, keep the balanced EXPLORE stage but also tune Logistic Regression hyperparameters (C/solver/penalty), then select the final model using the same criteria (LogLoss with an AUC floor) across all candidates. Finally, we should present outputs in the UI in a more decision-friendly way: show both the predicted winner’s Win% and the Home Win%, and add a simple confidence label (High/Medium/Low) so users can interpret results quickly and act consistently with the project’s decision-support goal.
